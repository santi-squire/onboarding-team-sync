The decision

  • Ramp the LaunchDarkly flag to 75% Flow B for one week, watch scope-leak + completion, then 100% and retire Flow A.
  • It's a flag % change, not a code change — the flows are built and live.
✓ What we got out of this

An email-first flow that prevents duplicate accounts at the door · −28.7pp less forgot-password friction · 80%+ of "abandoners" actually come back · and two methodology lessons: the new-device split is a sliding-window artifact (we anchor the cohort now), and accidental indie signups are a separate channel, not something this experiment moves (corrected with CS data).

What we changed so the last slip doesn't repeat

  • Design-first, dev-later. Lock the flow/spec before code — the login PR ballooned because the flow kept changing mid-build.
  • Product-decision checkpoints. Dedicated slots to clear open calls fast, so devs don't stall or rebuild.
  • Shorter, independent devs. PR + merge each piece on its own — no long stacked/cascaded branches.
  • Everything mapped in Jira (start + due + deps per ticket), cross-repo work tracked per repo — slippage is visible early.
What we learned about timing

We set dates before the flow was settled. Locking design and slicing the work smaller upfront is what makes targets real instead of optimistic.
