★ Support team confirmation

Zero accidental solo-barber cancellations in May.

Tristan asked McChesney Matro on the CS team to pull cancellation tickets. CS tags every cancellation with a reason. One of the tags is "Accidental Indy Self-Signup", used when a user contacts support to delete an account they signed up by mistake. April: 3 tickets with that tag. First 11 days of May (post full rollout): zero. The 6 other May cancellations were tagged Training, Out of Business, or no reason given — none accidental.

  • Snowflake (8wk baseline): ~13/wk before Apr 30 → ~3/wk after. That is a −77% drop, population-wide.
  • CS data: 3 accidental indy cancellations in April → 0 in May. Same direction, independent signal.
  • User base grew ~5% across the same window, so the drop is real, not "fewer users".
How we know someone signed up by accident
  • From our data (volume read): if a new solo-barber account uses the same email as a barber who's already attached to a shop, it's almost certainly a duplicate. The invited barber didn't realize they already had an account.
  • From support tickets (ground truth): the support team tags every cancellation ticket with a reason. "Accidental Indy Self-Signup" is one of those reason tags. No inference needed, the user literally told CS they signed up by mistake.
  • Support tickets are the cleanest signal. Our data shows the volume of likely duplicates; tickets show the ones users actually flagged as wrong.

★ How to read the two rows marked *

A wins by mechanics. B wins where it matters.

Both Flow A "wins" in the table above are measurement artifacts, not user wins. Here's why.

Why Flow A "wins" login abandoned
  • A has 1 screen to abandon. B has 2 (email step, then password step).
  • B mechanically fires more abandon events. Not a UX issue, it's a surface count.
  • 67% of B abandoners return later. Only 33% truly walk away.
Why Flow A "wins" per-exposure barber signup
  • B produces more raw barber signups (9.8% vs 7.2%).
  • But accidental indies are down 77% on the new flow.
  • So B has more signups but fewer are duplicates. The ones it catches look like genuine new barbers, not invited shop barbers mis-signing.
Where Flow B actually wins
  • Overall completion: +9pp B.
  • Welcome rate: +10pp B.
  • Forgot-password rate: −9pp B (less friction).
  • Duplicate-prevention: CS-confirmed 0 accidental signups in May.
What's next · mobile rebuild

Auth is moving to Descope

Auth is moving to Descope, a third-party login service. It can be configured to match either Flow A or Flow B, so the experiment conclusions still apply.

  • What changes: login backend and UI come from Descope, not custom code.
  • What doesn't: the visual flow we tested can be replicated inside Descope.
  • What we keep: the Flow B verdict carries over. We configure Descope as Flow B.
